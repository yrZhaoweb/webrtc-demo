# WebRTC Demo 多用户测试指南

本指南将帮助你全面测试 aves-core 和 aves-node 的功能。

## 📋 测试前准备

### 1. 构建依赖包

```bash
# 构建 aves-core
cd aves-core
npm install
npm run build

# 构建 aves-node
cd ../aves-node
npm install
npm run build
```

### 2. 启动服务器

```bash
cd ../webrtc-demo/server
npm install
npm run dev
```

你应该看到：

```
WebSocket server started on port 8080
🚀 Server is ready to accept connections
```

### 3. 启动 Web 客户端

在新的终端窗口：

```bash
cd webrtc-demo/web
npm install
npm run dev
```

访问显示的 URL（通常是 `http://localhost:5173`）

## 🧪 测试场景

### 场景 1: 基本双人聊天

**目标**: 验证基本的 WebRTC 连接和消息传输

**步骤**:

1. 打开两个浏览器窗口（或使用隐私模式）
2. 在窗口 A：
   - 点击"创建房间"
   - 输入名字"用户A"
   - 记录房间 ID
3. 在窗口 B：
   - 复制窗口 A 的邀请链接
   - 粘贴到地址栏访问
   - 输入名字"用户B"
4. 观察连接状态指示器（右上角）
5. 在两个窗口中互相发送消息

**预期结果**:

- ✅ 连接状态显示"已连接"（绿色）
- ✅ 对等连接状态显示 `connected`
- ✅ 数据通道状态显示 `open`
- ✅ 消息实时传输
- ✅ 用户列表正确显示
- ✅ 系统消息显示用户加入

**检查点**:

- [ ] 信令服务器连接成功
- [ ] WebRTC 连接建立成功
- [ ] 消息双向传输正常
- [ ] 用户列表更新正常
- [ ] 连接状态可视化正常

---

### 场景 2: 三人聊天室

**目标**: 验证多对等连接（Mesh 网络）

**步骤**:

1. 打开三个浏览器窗口
2. 窗口 A 创建房间（用户A）
3. 窗口 B 加入房间（用户B）
4. 等待 A-B 连接建立
5. 窗口 C 加入房间（用户C）
6. 观察连接状态：
   - 用户A 应该有 2 个对等连接（B 和 C）
   - 用户B 应该有 2 个对等连接（A 和 C）
   - 用户C 应该有 2 个对等连接（A 和 B）
7. 从任意窗口发送消息，验证所有人都能收到

**预期结果**:

- ✅ 每个用户都能看到其他 2 个用户
- ✅ 连接状态显示 2 个对等连接
- ✅ 消息广播到所有用户
- ✅ 所有连接状态都是 `connected`

**检查点**:

- [ ] Mesh 网络正确建立
- [ ] 消息广播功能正常
- [ ] 连接状态正确显示多个对等连接
- [ ] 性能表现良好

### 场景 3: 用户离开和重新加入

**目标**: 验证连接清理和重新建立

**步骤**:

1. 建立双人聊天（用户A 和 用户B）
2. 用户B 点击"离开房间"
3. 观察用户A 的界面：

   - 系统消息显示"用户B 已离开房间"
   - 对等连接状态应该清除
4. 用户B 重新加入同一房间
5. 验证连接重新建立

**预期结果**:

- ✅ 用户离开时连接正确清理
- ✅ 用户列表实时更新
- ✅ 系统消息正确显示
- ✅ 重新加入后连接重新建立

**检查点**:

- [ ] 离开房间清理逻辑正常

- [ ] 无内存泄漏

### 场景 4: 网络中断和重连

**目标**: 验证自动重连机制

**步骤**:

1. 建立双人聊天
2. 在浏览器开发者工具中：

   - 选择 "Offline" 模式
3. 观察连接状态变化：
   - 应该显示"重连中"（橙色）
4. 等待几秒后恢复在线：
   - 取消 "Offline" 模式
5. 观察自动重连过程

**预期结果**:

- ✅ 检测到断线
- ✅ 显示重连状态
- ✅ 自动尝试重连（最多 5 次）
- ✅ 重连成功后恢复正常

**检查点**:

- [ ] 断线检测正常

- [ ] 重连后功能正常

## 🐛 常见问题排查

### 问题 1: 连接状态一直显示"连接中"

**可能原因**:

- 信令服务器未启动
- WebSocket URL 配置错误
- 防火墙阻止连接

**排查步骤**:

1. 检查服务器是否运行：`curl http://localhost:8080`
2. 检查浏览器控制台错误
3. 检查 `.env` 文件配置

### 问题 2: WebRTC 连接失败

**可能原因**:

- ICE 候选收集失败
- STUN 服务器不可达
- 网络 NAT 类型不兼容

**排查步骤**:

1. 打开浏览器控制台查看 WebRTC 日志
2. 检查 `chrome://webrtc-internals/`（Chrome）
3. 尝试使用公共 STUN 服务器

### 问题 3: 消息发送失败

**可能原因**:

- DataChannel 未打开
- 连接已断开
- 消息过大

**排查步骤**:

1. 检查连接状态指示器
2. 查看数据通道状态
3. 检查浏览器控制台错误

## 🎯 性能基准

### 预期性能指标

- **消息延迟**: < 100ms（局域网）
- **连接建立时间**: < 3s
- **支持用户数**: 3-5 人（Mesh 网络）
- **消息吞吐量**: > 100 msg/s
- **内存使用**: < 100MB（单个客户端）
- **CPU 使用**: < 10%（空闲时）

### 测量工具

1. **浏览器开发者工具**
   - Performance 标签：CPU 和内存分析
   - Network 标签：网络延迟
   - Console：自定义性能日志

2. **Chrome WebRTC Internals**
   - 访问 `chrome://webrtc-internals/`
   - 查看详细的 WebRTC 统计信息

3. **服务器监控**
   ```bash
   # 监控服务器资源使用
   top -p $(pgrep -f "node.*index.ts")
   ```
